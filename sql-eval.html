Text-to-SQL · AI-Output Evaluation Sample

Evaluating an AI's SQL, like a second senior reviewer

Simon Sangla · 16 years in analytics (Lead Analyst, TripAdvisor) · SQL, dbt, Snowflake

A worked example of how I score a model's text-to-SQL output against a rubric: I identify each defect, cite where it breaks, score it, and write the correct query. Every judgement points to something observable, so a second rater can reproduce the score.

The task

Natural-language request → SQL

Prompt given to the model: "For each marketing channel, return the 7-day rolling average of revenue and the week-over-week change, over the last 90 days, excluding refunded orders."

Schema

-- one row per order
orders(order_id, channel, order_date date, revenue_cents int, status text)
       -- status in ('paid', 'refunded')
The model's answer

What the AI produced

SELECT channel,
       AVG(revenue_cents) OVER (PARTITION BY channel ORDER BY order_date
                                ROWS BETWEEN 7 PRECEDING AND CURRENT ROW) AS rolling_avg,
       revenue_cents - LAG(revenue_cents, 7) OVER (PARTITION BY channel ORDER BY order_date) AS wow_change
FROM orders
WHERE order_date > NOW() - 90          -- (no refund filter)
ORDER BY channel, order_date;

It runs and looks plausible. It does not answer the question.

The evaluation

Scorecard 2.0 / 5

Weighted across five criteria. Correctness is weighted highest because a wrong number is the costliest defect.

CriterionWeightScoreReason (observable)
Answers the question40%1Averages individual order amounts, not daily revenue per channel — wrong grain, so "rolling average of revenue" is not what is computed.
Window semantics25%2ROWS BETWEEN 7 PRECEDING spans 8 rows, and is row-based: on days with gaps or multiple orders it is neither 7 days nor daily.
Requirements met15%1The "excluding refunded orders" requirement is dropped — no status = 'paid' filter, so refunds inflate every figure.
Robustness10%2LAG(...,7) assumes 7 contiguous daily rows exist; a single missing day silently shifts the week-over-week comparison.
Portability / clarity10%3NOW() - 90 is dialect-fragile (integer vs interval); cents never converted; no CTE, so intent is hard to read.

Weighted total = 1.65, rounded to 2.0 / 5. Verdict: rejects the answer. The query is syntactically valid but answers a different question at the wrong grain.

The defects, ranked

What a senior reviewer flags

  • Critical  Wrong grain. Revenue must be aggregated to one row per channel per day before the rolling average; averaging raw order rows answers a different question.
  • Major  Missing requirement. No status = 'paid'; refunded orders are included against an explicit instruction.
  • Major  Window is row-based, not date-based. Use a RANGE frame over calendar days so gaps and multiple-orders-per-day behave correctly.
  • Minor  Gap-fragile WoW. LAG(...,7) should be a join to the same day one week earlier, so a missing day yields NULL rather than a wrong comparison.
  • Minor  Portability & units. NOW() - 90 → explicit interval; convert cents to currency; wrap in a CTE for readability.
The correction

What I would ship instead

-- Postgres / Snowflake. Aggregate to channel-day first, then window over calendar days.
WITH daily AS (
    SELECT channel,
           order_date                         AS d,
           SUM(revenue_cents) / 100.0          AS revenue   -- cents -> currency
    FROM orders
    WHERE status = 'paid'                       -- exclude refunds (requirement)
      AND order_date >= CURRENT_DATE - INTERVAL '90 days'
    GROUP BY channel, order_date              -- one row per channel-day
)
SELECT t.channel,
       t.d                                     AS day,
       t.revenue,
       AVG(t.revenue) OVER (
           PARTITION BY t.channel ORDER BY t.d
           RANGE BETWEEN INTERVAL '6 days' PRECEDING AND CURRENT ROW
       )                                       AS rolling_7d_avg,   -- 7 calendar days, gap-safe
       t.revenue - w.revenue                   AS wow_change         -- same weekday last week, NULL if absent
FROM daily t
LEFT JOIN daily w
       ON w.channel = t.channel
      AND w.d = t.d - INTERVAL '7 days'
ORDER BY t.channel, t.d;

Remaining assumption, stated honestly: the daily CTE has no row for a channel-day with zero paid orders, so the rolling window averages only days that exist. If the business wants zero-revenue days counted, join daily to a generated date spine per channel before the window. I would confirm that intent rather than guess it.
